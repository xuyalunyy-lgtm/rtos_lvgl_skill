# RTOS 软件架构设计提示词（嵌入式）

用途：在 FreeRTOS / 嵌入式项目中产出可落地的架构方案、模块划分与验收项。

## 先决条件（先确认，缺一不可）
1. 产品目标与验收指标（吞吐、时延、功耗）
2. 平台约束（SoC、外设、总线、DMA/IRQ/RTOS 版本）
3. 实时约束（上限抖动、采样间隔、最大丢包）
4. 资源边界（RAM/FLASH/栈/队列深度）
5. 稳定性要求（WDT、错误恢复、回退策略）

缺失任一项时，只输出补充问题清单，不直接设计。

## 设计输出必须覆盖

### 1. 输入-处理-输出（I/P/O）三层 + 消息队列解耦
- 输入层（Input/Inbound）：仅采集裸数据并打包入队，不做业务计算。  
  - 数据来源：传感器、按键、RF 接收、UART/SPI 等  
  - 常见实现：ISR/DMA 回调->RawMessage->`input_queue`
- 核心层（Core/Business）：阻塞等待输入队列，完成滤波、状态机、协议解析。  
  - 处理完成后输出“控制指令”到 `output_queue`
- 输出层（Output/Outbound）：只负责执行，不做业务决策。  
  - 动作：电机、显示、蓝牙、Flash 写入、上报  
- 约束：
  - 每一层只依赖下游的“数据合约”
  - 定义完整数据模型（time_stamp、source、payload_len、crc、seq）
  - 明确所有权与释放责任（谁分配、谁释放）
  - 给出队列深度上限、溢出策略（丢旧/丢新/告警）

### 2. 状态机驱动架构（FSM / HFSM）
- 采用事件驱动状态机控制系统行为（建议 HFSM 管理复杂子域）。
- 输出：
  - 主状态表：`STATE x EVENT -> GUARD -> ACTION -> NEXT_STATE`
  - 必要时拆分子状态机（如业务态、通信态、恢复态）
  - 明确错误态与恢复流程（例如 `ERROR -> RECOVERING -> IDLE`）
  - 明确禁止行为：不允许 ISR 直接改业务状态，不允许跨层“读改写共享全局”

### 3. 硬件抽象层（HAL）+ 组件化
- 所有外设访问通过 `ops` 接口结构体，应用层只认识接口，不看 SDK 细节。
- 最小形式：
  - `xxx_ops_t`：全部函数指针（init/open/start/stop/read/write/ioctl/deinit）
  - `xxx_handle_t`：持有 context + ops + 后端状态
  - `xxx_component_t`：可插拔实现（同一接口，支持不同芯片/驱动替换）
- 输出：
  - 至少 5 个关键 HAL 接口示例（如 adc、gpio、i2c、display、storage）
  - 平台映射表（Interface -> 实现文件 -> 编译宏/链接策略）
  - 芯片替换时不改 core 层的清单

## 必须给出的交付结构（固定）

### A. 架构图（文本）
- 数据流：`IRQ/DMA -> Input -> Core -> Output`
- 控制流：`Event/Timer -> CoreFSM/HFSM -> Output commands`

### B. 任务与队列
- 每个 task 的优先级、栈、阻塞点
- 每条队列的元素大小/深度/丢弃策略

### C. 模块表
- Module | 职责 | 输入 | 输出 | 生命周期 | 依赖

### D. 状态机规范
- 状态集、事件集、转移表、错误恢复表

### E. HAL 与组件
- 接口定义草案与平台实现映射

### F. 风险与验收
- 高峰中断、队列风暴、内存碎片、栈超限、看门狗触发
- 建议的观测点（计数器、丢包数、最大排队深度、状态切换计数）
