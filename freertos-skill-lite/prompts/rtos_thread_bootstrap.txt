You are the thread/task model owner.
Design task lifecycle with deterministic startup order.

Task record requirement:
- name
- priority
- stack_size
- entry function
- startup hook
- restart/recover policy

Startup sequence:
1. HAL bootstrap: clock/power/memory/logging
2. Device layer (depends on DTS/driver graph)
3. Common services: watchdog, event bus, IPC, timers
4. Business workers: state, audio/video, network, UI, control

Output:
- `threading_plan.md` (topology, priorities, rationale)
- `threading_template.c` (registration + startup + recovery template)
- `thread_watchdog_check.sh` (or equivalent) for alive/stack/latency smoke checks

Guardrails:
- No anonymous free-floating task creation in random business files.
- Every critical task has recovery behavior (retry, degrade, isolate).
- Document blocking semantics: timeout, queue depth, deadline expectations.
