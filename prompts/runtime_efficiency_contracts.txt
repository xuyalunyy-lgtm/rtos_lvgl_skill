# RTOS 运行时效率契约专精提示词

> 约束 ID：C29.1-C45.5 → [constraint_detail.md](../references/constraint_detail.md)

## 适用场景

新增模块、工程 review、架构 review、死机/卡顿/不可复现问题定位、多人维护项目接手。

---

## C29 — 模块契约（Module Contract）

每个非平凡模块必须在 `.h` 或模块头注释声明：

- 线程上下文：哪些 API 可从 task / ISR / timer callback / LVGL thread 调用
- 阻塞语义：是否阻塞、最大等待时间、是否可重入
- 所有权：入参/出参/回调/Queue payload 由谁 alloc/free/release
- 生命周期：`init -> start -> stop -> deinit` 的合法顺序
- 错误语义：返回码、可恢复错误、不可恢复错误

```c
/* module_contract:
 * context: task only; ISR must use sensor_notify_from_isr()
 * blocking: read <= 20ms; stop <= 50ms
 * ownership: caller owns config; module owns handle after init
 * lifecycle: init -> start -> stop -> deinit
 * errors: DEVICE_ERR_TIMEOUT is recoverable
 */
```

### C29.6-C29.10 - Module boundary contract

Every non-trivial module must keep one primary reason to change. Declare the
boundary in the `.h` file or module header before implementation:

```c
/* module_boundary:
 * responsibility: collect sensor samples and publish normalized events
 * public_api: sensor_init, sensor_start, sensor_stop, sensor_get_status
 * dependencies: i2c_bus, app_event_bus
 * forbidden_dependencies: lvgl, network_wss, storage_nvs
 * events_in: SENSOR_CMD_START, SENSOR_CMD_STOP
 * events_out: SENSOR_EVT_SAMPLE_READY, SENSOR_EVT_FAULT
 * owned_resources: i2c0, sensor_task, sensor_queue
 */
```

Rules:
- Keep cohesion high: one module owns one business capability or hardware resource family.
- Keep coupling low: cross-module communication goes through public API, queue/event, or callback contracts.
- Do not include another module only to reach into its private structs or globals.
- Do not share writable global context across modules; pass handles, descriptors, or events.
- Declare forbidden dependencies when a lower layer must never call a higher layer, such as driver -> UI or network -> LVGL.

## C30 — 任务/队列拓扑表

工程级 review 或新增多任务模块时，先输出拓扑表，再写实现。

| Task | Priority | Stack | Queue | Producers | Consumers | Timeout | Backpressure | Exit |
|------|----------|-------|-------|-----------|-----------|---------|--------------|------|
| `sensor_task` | high-2 | 3072 | `sensor_q:8` | ISR, timer | fusion | 20ms | drop-oldest + count | stop flag |

必须说明：

- 每条 Queue 的元素类型、payload 所有权和满队列策略
- 每个 task 的退出条件，禁止只能 reboot 才退出
- 高频 producer 的背压策略，禁止无限扩容

## C31 — 超时预算

所有等待都必须有“为什么等、最多等多久、失败后如何恢复”。

禁止默认写：

```c
xQueueReceive(q, &evt, portMAX_DELAY);
```

优先写：

```c
if (xQueueReceive(q, &evt, pdMS_TO_TICKS(20)) != pdPASS) {
    stats.rx_timeout++;
    continue;
}
```

允许永久等待的例外必须同时满足：

- 只在 dedicated idle/daemon consumer 中使用
- 注释说明不会持有 mutex、不会阻塞 stop/deinit
- 有 watchdog/health counter 能证明任务仍活着

---

## C32 — 可观测性优先

关键链路至少保留：

- 当前状态与最近错误：`state` / `last_error` / `last_error_line`
- 计数器：timeout/drop/retry/reconnect/overflow/underrun
- 资源水位：stack high-water mark、heap free/min/largest、queue high-water mark
- 阶段耗时：init/connect/handshake/decode/render/flush max time
- 低频 dump：现场日志能还原“最后 10 个关键事件”

遥测必须低频、脱敏、可关闭；热路径只加 counter，不直接打印。

---

## C33 — 生命周期对称

凡是出现以下动作，必须能找到对应收尾：

| Acquire | Release |
|---------|---------|
| `init/open/start/enable` | `stop/disable/close/deinit` |
| `alloc/create/register/attach` | `free/delete/unregister/detach` |
| `power_on/clock_enable/dma_start` | `dma_stop/clock_disable/power_off` |

多资源函数必须走统一 `cleanup:`，异常路径和正常路径共用同一批 release helper。

---

## C34 — 热路径禁区

以下上下文属于 hot path：

- ISR / DMA callback / HAL complete callback
- LVGL flush / timer callback
- audio/video frame process / encode / decode / render
- 控制环、采样循环、网络收包回调

hot path 禁止：

- 动态分配/释放：`malloc/free/pvPortMalloc/vPortFree`
- 重日志：`printf/LOG_*` 高频打印
- 阻塞等待：`portMAX_DELAY`、无界 socket/TLS/file IO
- 重型解析/创建：cJSON parse、codec create/open、TLS handshake
- 直接跨层调用：UI/网络/文件系统/业务状态机重活

正确做法：只 `notify/enqueue/set flag/increment counter`，重活放到普通 task。

---

## C35 — 关键路径预算表
任何启动、联网、音频、视频、UI、OTA、低功耗唤醒等关键路径，都必须先给预算表，再实现。

| Path | Stage | Budget | Timeout | Fallback | Metric |
|------|-------|--------|---------|----------|--------|
| boot | mount fs | 120ms | 300ms | factory defaults | `boot_fs_max_ms` |
| audio | capture -> encode | 10ms/frame | 15ms | drop frame | `audio_encode_max_us` |

必须声明：
- 每个阶段的 owner、最大耗时、超时处理、降级策略
- 串行阶段是否可并行化，禁止无证据地在关键路径做长 IO
- 阶段 max time 与 timeout/drop/retry counter

---

## C36 — 数据拷贝预算
跨 task、跨核、DMA、网络、音视频 frame、LVGL draw buffer 等路径，必须声明数据移动策略。

优先级：
1. descriptor/index/handle 传递
2. bounded one-copy
3. 明确允许的 deep copy

禁止：
- 默认传大结构体进 Queue
- 每帧 malloc + memcpy + free
- DMA buffer 未说明 cache clean/invalidate 与 ownership transfer

必须给出 `copy_count`、buffer owner、release 方和满池策略。

---

## C37 — 背压与降级策略
所有高频 producer、网络收包、音视频帧、传感器采样、日志、UI 事件队列都必须有背压策略。

可选策略必须写明触发条件：
- drop-oldest / drop-newest / coalesce / overwrite
- 降采样 / 降帧率 / 降码率 / 暂停非关键任务
- bounded retry / circuit breaker / graceful degradation

禁止满队列后无限等待，禁止只扩队列深度而不说明 RAM 与延迟代价。

---

## C38 — 故障隔离与自动恢复
关键模块必须定义故障域、恢复动作和重试上限。

必须声明：
- task / driver / protocol / cloud / UI 的故障边界
- recoverable 与 fatal 错误分类
- retry backoff、最大重试次数、熔断时间
- supervisor/watchdog/health counter 如何发现卡死
- 恢复失败后的降级模式或安全停机动作

---

## C39 — 配置矩阵约束
Kconfig、feature flag、板级差异、SDK 差异、外设选择必须收敛为配置矩阵。

必须声明：
- feature -> dependency -> resource -> default -> test mode
- 每个 `#ifdef` 属于 platform / board / feature / debug 哪一类
- 禁止业务逻辑中散落未命名 feature macro
- 配置变化必须同步 bring-up checklist 与回归样本

---

## C40 — 一键复现闭环
每个复杂问题、bring-up 路径或新模块必须留下可复现命令。

最少包含：
- build / flash / monitor / log capture
- crash decode / core dump decode / symbol path
- 最小配置、测试入口、预期输出
- 失败日志保存位置和脱敏规则

目标是让下一个工程师无需重新问“怎么跑起来”。

---

## C41 — 回归样本优先
新约束、新 checker、新 bugfix 必须优先沉淀 good/bad 样本或最小复现片段。

必须满足：
- bad 样本能稳定触发问题或审查项
- good 样本展示推荐修复模式
- 样本不绑定具体产品业务，不包含密钥、客户名、私有路径
- checker 或人工 checklist 明确引用样本

---

## C42 — 板级资源契约
GPIO、DMA、clock、IRQ priority、cache、heap kind、PSRAM、外设复用必须在模块接入前声明。

必须声明：
- board resource owner 与冲突检测方式
- DMA/cache 对齐、cacheable/non-cacheable、heap capability
- IRQ priority、ISR-safe API、跨核访问边界
- clock/power domain 生命周期与低功耗约束
- 板级差异如何进入 platform/profile，而不是散落在业务代码

---

## C43 — 锁预算与优先级反转防护
所有 mutex、recursive mutex、跨模块锁和共享资源 guard 都必须有等待预算、持锁预算和优先级反转防护。

必须声明：
- 锁等待 timeout、最大持锁时间、超时后的恢复或降级动作
- 持锁期间只做短临界区状态读写，禁止网络/TLS/Flash/file IO/cJSON parse/delay/codec open
- 保护共享资源使用带优先级继承的 mutex；binary semaphore 只用于事件同步，不作为 mutex
- 多锁嵌套必须声明全局 `lock_order` / `lock_rank`
- ISR/callback/LVGL flush/audio/video hot path 禁止拿 mutex，只能 notify/enqueue/set flag/counter

---

## C44 — 临界区/关中断预算
所有 `taskENTER_CRITICAL`、`portDISABLE_INTERRUPTS`、`__disable_irq`、平台 IRQ mask 或全局中断屏蔽路径都必须短小、对称、可解释。

必须声明：
- `critical_budget` / `irq_off_budget` / `max_irq_off_us`，并说明为什么必须关中断
- 关中断期间只做寄存器、flag、counter、短状态切换
- 禁止阻塞等待、动态分配、日志、`memcpy` 大拷贝、Flash/file IO、cJSON parse、codec create/open
- 每个 enter/disable 都必须保证 exit/enable；错误路径用 cleanup 或单出口恢复
- ISR/callback/LVGL flush/audio/video hot path 不制造长临界区；复杂一致性交给 task/mutex/队列/双缓冲

---

## C45 — 传感器集成契约
I2C/SPI/ADC/IMU/环境类传感器接入必须先把 datasheet、总线超时、数据就绪、单位与校准生命周期写清楚，再写采样逻辑。

必须声明：
- init/probe 的 register map 依据、`WHO_AM_I` / `chip_id` 校验和不匹配处理
- 每个 I2C/SPI transaction 的 timeout、retry/backoff、错误分类和 telemetry counter
- data-ready/DRDY/status/fifo 等待策略：中断/notify 优先，有界轮询必须有 deadline
- sample 结构必须包含 timestamp、unit、range、scale/offset 或 calibration version
- calibration/self-test/warm-up 放在 start 或低频维护任务，禁止塞进采样 hot path

---

## Agent Checklist

- [ ] C29 模块头是否声明 context/blocking/ownership/lifecycle/error？
- [ ] C30 task/queue 拓扑是否能解释 producer/consumer/timeout/backpressure/exit？
- [ ] C31 所有等待是否有有限 timeout，永久等待是否有例外说明？
- [ ] C32 是否有状态、错误、计数器、水位、阶段耗时与低频 dump？
- [ ] C33 acquire/release 是否对称，异常路径是否统一 cleanup？
- [ ] C34 hot path 是否只做轻量事件投递和计数？
- [ ] C35 关键路径是否有 stage/budget/timeout/fallback/metric 表？
- [ ] C36 数据移动是否声明 copy count、owner、release 和 DMA cache 策略？
- [ ] C37 队列/帧/日志/网络是否有背压与降级策略？
- [ ] C38 故障域、retry/backoff、supervisor 与安全降级是否明确？
- [ ] C39 Kconfig/feature/board/SDK 差异是否收敛为配置矩阵？
- [ ] C40 build/flash/log/decode/test 是否可一键复现？
- [ ] C41 新约束或 bugfix 是否有 good/bad 回归样本？
- [ ] C42 GPIO/DMA/clock/IRQ/cache/heap 等板级资源契约是否声明？
- [ ] C43 锁等待、持锁时间、lock_order、优先级继承和热路径禁锁是否明确？
- [ ] C44 临界区/关中断是否短小、对称、有预算，且无阻塞/分配/日志/大拷贝？
- [ ] C45 传感器 init/probe、总线 timeout、data-ready、sample metadata 和 calibration lifecycle 是否明确？
