# Display Driver Safety Prompt

> Constraint IDs: C23.1-C23.6. See [constraint_detail.md](../references/constraint_detail.md).

## Scope

Use this prompt for LCD/OLED display driver bring-up, LVGL display porting, backlight control, frame-buffer sizing, tearing prevention, and production UI performance reviews.

### 1. C23.1 LCD initialization sequence

- Follow the panel datasheet exactly for reset pulse width, reset-release delay, sleep-out delay, display-on delay, and command spacing.
- Keep reset, power rails, SPI/RGB/DSI bus init, and orientation setup in a single auditable sequence.
- Log the panel ID, timing profile, resolution, color format, and interface mode during boot.

```c
void lcd_init(void)
{
    gpio_set_level(LCD_RST_PIN, 0);
    vTaskDelay(pdMS_TO_TICKS(10));
    gpio_set_level(LCD_RST_PIN, 1);
    vTaskDelay(pdMS_TO_TICKS(120));

    lcd_send_cmd(0x11);  /* Sleep Out */
    vTaskDelay(pdMS_TO_TICKS(120));

    lcd_send_cmd(0x29);  /* Display On */
    vTaskDelay(pdMS_TO_TICKS(20));

    lcd_set_orientation(LCD_ORIENTATION_LANDSCAPE);
}
```

Anti-patterns:
- Sending `Sleep Out` immediately after reset release.
- Hiding panel timing in board-specific magic delays without comments.
- Initializing LVGL before the panel bus and framebuffer ownership are stable.

### 2. C23.2 Backlight PWM and power control

- Prefer PWM brightness control over direct GPIO on/off control.
- Disable the backlight power enable pin when brightness is zero if the board supports it.
- Fade brightness changes that are visible to users; keep emergency blanking immediate.

```c
void lcd_set_brightness(uint8_t percent)
{
    if (percent == 0) {
        ledc_set_duty(LEDC_LOW_SPEED_MODE, LEDC_CHANNEL_0, 0);
        ledc_update_duty(LEDC_LOW_SPEED_MODE, LEDC_CHANNEL_0);
        gpio_set_level(LCD_BL_EN_PIN, 0);
        return;
    }

    gpio_set_level(LCD_BL_EN_PIN, 1);
    uint32_t duty = ((uint32_t)percent * 8191U) / 100U;
    ledc_set_duty(LEDC_LOW_SPEED_MODE, LEDC_CHANNEL_0, duty);
    ledc_update_duty(LEDC_LOW_SPEED_MODE, LEDC_CHANNEL_0);
}
```

Review points:
- PWM frequency is high enough to avoid visible flicker and audio coupling.
- Low-power mode turns off both PWM duty and panel/backlight enable where required.
- Brightness state survives suspend/resume without flashing full white.

### 3. C23.3 Refresh rate, tick, and VSync budget

- Match `lv_timer_handler()` cadence to panel refresh and UI workload; do not spin at 1 ms without evidence.
- Keep display flush asynchronous when the platform supports DMA or LCD engine callbacks.
- Track frame time, flush time, dirty area, and dropped/merged updates.

```c
#define LCD_REFRESH_RATE_HZ 60U
#define LVGL_TICK_PERIOD_MS (1000U / LCD_REFRESH_RATE_HZ)

static void lvgl_task(void *arg)
{
    for (;;) {
        uint32_t next_ms = lv_timer_handler();
        if (next_ms == 0 || next_ms > LVGL_TICK_PERIOD_MS) {
            next_ms = LVGL_TICK_PERIOD_MS;
        }
        vTaskDelay(pdMS_TO_TICKS(next_ms));
    }
}
```

Common targets:
- 60 Hz panel: 16 ms frame budget.
- 30 Hz panel or low-power page: 33 ms frame budget.
- UI task CPU should leave headroom for audio/network/IO tasks.

### 4. C23.4 Tearing protection

- Prefer panel TE signal, VSYNC interrupt, or double buffering for full-screen updates.
- Single-buffer direct writes are acceptable only for small dirty areas or non-animated screens verified on hardware.
- Do not call `lv_disp_flush_ready()` before the active transfer is actually safe to reuse.

```c
void lcd_flush_cb(lv_disp_drv_t *drv, const lv_area_t *area, lv_color_t *color_p)
{
    lcd_dma_submit(area, color_p);
    /* DMA completion ISR/task calls lv_disp_flush_ready(drv). */
}
```

Anti-patterns:
- CPU memcpy inside the flush hot path while holding the LVGL mutex.
- Reusing draw buffers before DMA completion.
- Mixing video overlay teardown with LVGL framebuffer close/reopen.

### 5. C23.5 Framebuffer and draw-buffer management

- Size buffers from resolution, color depth, RAM/PSRAM class, DMA alignment, and cache policy.
- Check every allocation and provide a partial-refresh fallback.
- Keep DMA-visible buffers aligned and in memory that the display peripheral can read.

```c
#define PARTIAL_LINES 40U
static lv_color_t draw_buf1[LCD_WIDTH * PARTIAL_LINES] __attribute__((aligned(32)));
static lv_color_t draw_buf2[LCD_WIDTH * PARTIAL_LINES] __attribute__((aligned(32)));

void lvgl_display_buffer_init(void)
{
    static lv_disp_draw_buf_t draw_buf;
    lv_disp_draw_buf_init(&draw_buf, draw_buf1, draw_buf2, LCD_WIDTH * PARTIAL_LINES);
}
```

Sizing rule of thumb:
- RGB565 full screen bytes = `width * height * 2`.
- Double full-screen buffer doubles that number.
- On small RAM systems, prefer partial refresh such as 1/5 or 1/10 screen height.

### 6. C23.6 LVGL display driver registration

- Register draw buffer, flush callback, resolution, color format, and rotation in one place.
- For LVGL v8/v9 differences, keep a narrow porting layer instead of scattering version checks through app code.
- Validate linkability of deinit APIs on the target SDK before promising full global teardown.

```c
void lvgl_port_init(void)
{
    static lv_disp_drv_t disp_drv;
    lv_disp_drv_init(&disp_drv);
    disp_drv.draw_buf = &g_draw_buf;
    disp_drv.flush_cb = lcd_flush_cb;
    disp_drv.hor_res = LCD_WIDTH;
    disp_drv.ver_res = LCD_HEIGHT;
    disp_drv.full_refresh = 0;
    lv_disp_drv_register(&disp_drv);
}
```

Review points:
- `hor_res`/`ver_res` match physical rotation and touch coordinates.
- Flush callback always calls `lv_disp_flush_ready()` on every success/failure path.
- Display teardown stops UI timers/tasks before freeing buffers or closing framebuffers.

### 7. Layout and style performance

- Reuse global `lv_style_t` objects; avoid repeated runtime style allocation on every page switch.
- Keep layout hierarchy shallow. For static pages, absolute positions can be cheaper than nested Flex/Grid trees.
- Avoid software-heavy effects such as alpha blending, shadows, blur, and canvas drawing on low-end MCUs unless measured.

```c
static lv_style_t common_style;

void init_style(void)
{
    lv_style_init(&common_style);
    lv_style_set_bg_color(&common_style, lv_color_hex(0x000000));
    lv_style_set_text_font(&common_style, &lv_font_montserrat_14);
}
```

### 8. Widget events and input debounce

- Input drivers should debounce and rate-limit raw touch/key events before they reach business logic.
- Page navigation callbacks should post an event and return; heavy page creation must run on the UI task boundary.
- During media playback, page transitions, or async reload, guard repeated clicks with an explicit state machine.

```c
if (g_page_state != PAGE_IDLE) {
    LOG_W("click ignored during transition state=%d", g_page_state);
    return;
}
post_ui_nav_event(NAV_NEXT_PAGE);
```

### 9. Compiler, RTOS, and DMA alignment

- Build LVGL and display port sources with optimized flags appropriate for the target project.
- Keep `lv_timer_handler()` in a medium-priority task; never perform blocking network or file IO in that task.
- Apply cache clean/invalidate rules around DMA transfers on platforms with non-coherent caches.

```c
__attribute__((aligned(32))) static lv_color_t framebuf[BUFFER_SIZE];
```

### 10. Low-power and dirty refresh

- Use the return value of `lv_timer_handler()` to extend sleep when the UI is idle.
- Keep clock/second updates dirty-area based; avoid full-screen invalidation for small labels.
- Stop nonessential animations before entering low power and restore them explicitly on wake.

```c
uint32_t next_ms = lv_timer_handler();
if (next_ms > 100U) {
    enter_low_power_wait(next_ms);
}
```

### 11. Page lifecycle and navigation stack

- Create normal pages under the provided content parent; do not create a screen and later reparent it.
- Own timers, allocations, image descriptors, and media handles at the page/container level and release them from delete hooks.
- Root pages need explicit back-boundary handling to avoid empty stack or black screen states.

Checklist:
- [ ] No duplicate push caused by double click or repeated async callback.
- [ ] Every page-local allocation has one owner and one release path.
- [ ] Cached pages restore shared background and resource handles before becoming visible.

### 12. Resource, image, and media overlay lifecycle

- Put all background/icon access behind the common resource layer so cache policy and fallback are consistent.
- Decode large JPEG/PNG backgrounds outside the touch callback path where possible.
- For short video overlays, keep LVGL and its primary framebuffer alive; render video on a dedicated top layer when the SDK supports it.

Required logs:
- `page transition begin/end`
- `asset decode begin/end`
- `page cache hit/store/evict`
- `media done, ui nav requested`
- `ui nav done`

### 13. Agent checklist

- [ ] C23.1 LCD reset and command timing follows the datasheet.
- [ ] C23.2 Backlight uses PWM and has a real low-power off path.
- [ ] C23.3 `lv_timer_handler()` cadence matches panel refresh and CPU budget.
- [ ] C23.4 Tearing prevention and DMA completion ownership are explicit.
- [ ] C23.5 Framebuffer size, memory class, allocation failure, and cache policy are reviewed.
- [ ] C23.6 LVGL display driver registration and teardown paths are complete.
- [ ] Page navigation is async, guarded, and single-owner.
- [ ] Display/media/resource logs can explain low FPS, black screen, and reboot-after-video cases.
### 14. 生产级策略（补充）

以上 1~13 条更偏优化手法，下面是可落地的生产化策略（10 项）：

1. 输入与显示链路解耦
   - 原则：UI 线程只处理“最终渲染状态”，数据采集在前端任务去抖后异步入队，显示任务消费。
   - 示例：将传感器更新转为消息事件，LVGL 周期处理合并后的最新状态。

```c
typedef struct {
    int32_t v_batt_mv;
    int32_t temp_c;
    uint8_t flags;
} ui_msg_t;

static QueueHandle_t ui_queue;
static ui_msg_t g_ui_state;

void sensor_task(void *arg)
{
    ui_msg_t msg;
    for (;;) {
        read_sensor_msg(&msg);              // 原始传感器读取
        xQueueOverwrite(ui_queue, &msg);    // 仅保留最新值，避免堆积
    }
}

void ui_commit_task(void *arg)
{
    ui_msg_t msg;
    for (;;) {
        if (xQueueReceive(ui_queue, &msg, pdMS_TO_TICKS(20))) {
            lv_async_call(apply_ui_msg, pvPortMallocClone(&msg));
        }
    }
}
```

2. 统一定时器调度
   - 原则：系统只要有一个主 Tick/定时源，统一驱动 `lv_timer_handler()`、动画、轮询超时和低功耗判定。
   - 示例：用一个 1ms~10ms 的主任务周期计算下次唤醒时间。

```c
void lvgl_main_task(void *arg)
{
    for (;;) {
        uint32_t delay_ms = lv_timer_handler();
        if (delay_ms < 1) delay_ms = 1;
        if (delay_ms > 20) vTaskDelay(pdMS_TO_TICKS(20));
        else vTaskDelay(pdMS_TO_TICKS(delay_ms));
    }
}
```

3. 大列表/长列表虚拟化
   - 原则：超过可见项阈值的列表，采用“按需创建+复用”而不是一次性创建全量控件。
   - 典型手段：固定容器 + 复用对象池（pageing / windowed list），滚动时重绑定数据。

```c
// 每次仅创建 N+1 个可见行，滚动时复用 row 对象
for (int i = 0; i < VISIBLE_ROWS; i++) {
    bind_row(rows[i], data_base_index + i);
}
```

4. 精细脏区更新
   - 原则：避免 `lv_obj_invalidate(lv_scr_act())` 全屏刷新，尽量更新具体对象或最小矩形。
   - 频闪场景（秒表/计时）建议只刷新数字区块，不刷新底纹与容器边框。

```c
lv_area_t a = { .x1 = x, .y1 = y, .x2 = x + w, .y2 = y + h };
lv_obj_invalidate_area(lv_scr_act(), &a);
```

5. 资源与图像加载分层
   - 原则：资源层（图标/字体）与业务层分离，按场景懒加载，避免“首页进来就加载全部素材”。
   - 建议把高频资源预加载到 SRAM，低频资源放外部 flash，按需异步拉取并回收。

```c
if (need_icon_big_chart) {
    lv_img_set_src(img, &img_large_chart); // 热路径
} else {
    lv_img_set_src(img, &img_small_chart_placeholder);
    defer_load_icon(ICON_BIG_CHART);       // 冷路径异步替换
}
```

6. 渲染与 I/O 任务分离
   - 原则：SPI/DBI 发送、DMA 完成回调只做最小工作（提交 / 信号），不要在回调里做布局、创建对象和日志输出。
   - 回调只更新状态位，真正的 UI 状态机放到主任务处理。

```c
volatile bool g_lcd_flush_done = false;
void lcd_dma_done_isr(void)
{
    g_lcd_flush_done = true;
    BaseType_t hpw = pdFALSE;
    vTaskNotifyGiveFromISR(lvgl_task_handle, &hpw);
}
```

7. 长耗时布局分片
   - 原则：复杂布局/主题切换时按帧分解，避免一个 tick 内创建几十上百个控件导致卡顿超时。
   - 示例：用队列驱动“每帧创建一批控件”。

```c
static int pending_steps;
void ui_work_timer_cb(lv_timer_t *t)
{
    int budget = 0;
    while (budget++ < 8 && pending_steps-- > 0) {
        create_one_widget_chunk();
    }
    if (pending_steps <= 0) lv_timer_del(t);
}
```

8. 低内存降级模式
   - 原则：检测到内存紧张时自动降级为“轻量渲染模式”：关闭 perf/部分动画、切换到单缓冲、禁用模糊/阴影。
   - 可定义一个运行时阈值，超限后动态应用。

```c
if (lv_mem_monitor.used_pct > 85 || lv_mem_test_fragmentation() > 40) {
    set_ui_profile(QOS_LOW_MEM);   // 缩减缓冲区 + 简化主题 + 禁用动态效果
}
```

9. 黑屏恢复与异常保护
   - 原则：加入 UI 看门狗与最后画面快照；若 2~3 个周期未进入 `lv_task_handler` 或 flush 挂住，执行恢复流程。
   - 恢复策略：重建显示栈、重新注册驱动、回退到安全首页并记录错误码。

```c
if (ticks_since_last_flush > 300) {
    lv_deinit();
    init_display_hw();
    init_lvgl();
    lv_scr_load(home_screen);
}
```

10. CI 性能回归门禁（轻量）
   - 原则：把关键指标固化到测试脚本，任何提交要稳定通过：
     - 启动时间
     - 空闲画面 FPS
     - 最大 `lv_timer_handler()` 耗时
     - 内存峰值/碎片度
     - 关键界面加载失败率
   - 示例：在 CI 里加固定场景脚本，对应阈值超限则失败。

```bash
python tools/lvgl_frame_rate_checker.py --dir ./src
```
该命令只做提交前的静态预检；FPS、flush 时延和内存碎片的回归阈值必须由目标工程的设备或模拟器测试入口采集并比较。

### 14.1 生产级检查清单
- [ ] 关键输入事件是否去抖 + 入队，UI 线程无阻塞地消费？
- [ ] 是否存在 `lv_timer_handler()` 抖动：长尾耗时、空闲阻塞、睡眠唤醒逻辑可复现？
- [ ] 大列表是否只创建可见项？有无对象池回收复用？
- [ ] 刷新是否尽量限制到变更区域，而不是每帧全屏 invalidate？
- [ ] 资源加载是否按场景分层，是否有内存峰值上限与卸载策略？
- [ ] DMA 回调是否只做轻量信号，业务逻辑是否移动到任务上下文？
- [ ] 复杂页面构建是否支持分片，单帧创建控件数有上限？
- [ ] 低内存模式开关是否存在并可回退，不会卡住动画与事件？
- [ ] 是否有显示异常/flush 超时自恢复路径并写入日志？
- [ ] CI 是否固定跑 1~2 套可复现实验，并对 FPS、内存、闪退门槛给出 fail-fast 条件？


### 15. 进阶生产策略（10项）

1. 多分辨率与 DPI 自适配
- 原则：用统一 scale 体系避免“不同屏幕上重排风暴”。字体、间距、圆角统一走缩放策略。
- 示例：基于 LVGL 逻辑像素扩展布局。

```c
#define UI_BASE_DPI (160)
static int32_t ui_scale;

static inline uint16_t ui_scale_px(int16_t px)
{
    return (int16_t)((px * ui_scale) / 100);
}

void ui_scale_init(lv_disp_t *d)
{
    uint16_t dpi = lv_disp_get_dpi(d);
    ui_scale = (dpi > 0) ? (int32_t)dpi * 100 / UI_BASE_DPI : 100;
    lv_obj_set_style_pad_all(lv_scr_act(), ui_scale_px(8), LV_PART_MAIN);
}
```

2. 主题与控件变体集中管理
- 原则：颜色/间距/边框统一在 theme 里配置，运行时只做“按主题替换”，避免散落更新。
- 示例：启动阶段一次性初始化 theme。

```c
static lv_theme_t *app_theme;
void init_ui_theme(void)
{
    app_theme = lv_theme_default_init(lv_disp_get_default(), lv_color_hex(0x1e1e2a),
                                     lv_color_hex(0x00a8ff), false, lv_font_default);
    lv_disp_set_theme(lv_disp_get_default(), app_theme);
}
```

3. 帧预算化（Frame Budget）
- 原则：给每帧设置时间预算（如 16/33ms），把大批量更新拆帧。
- 示例：超预算时只执行低优先级更新。

```c
static uint32_t budget_ms = 16;
if (update_cost_ms > budget_ms) {
    defer_noncritical_jobs();  // 下个 tick 再做，不阻塞当前帧
} else {
    do_full_ui_refresh();
}
```

4. 长按/连击防重入
- 原则：按钮点击去抖与幂等保护，避免同一路径被连续触发。
- 示例：150ms 内忽略重复点击。

```c
static uint32_t last_click_ms = 0;
static void on_btn_click(void *user_data)
{
    uint32_t now = lv_tick_get();
    if (now - last_click_ms < 150) return;
    last_click_ms = now;
    open_page(user_data);
}
```

5. 资源冷热路径双轨制
- 原则：热资源（首屏）提前预热，冷资源（非首屏）按需加载并支持失败回退。
- 示例：占位图 + 异步替换。

```c
lv_img_set_src(img, &img_placeholder);
request_async_load(IMG_ID_SPEED, img, on_img_ready);
// on_img_ready 成功则替换，否则保留 placeholder 并计数告警
```

6. 异常兜底界面（Fallback UI）
- 原则：任一子系统异常时，快速回退到轻量错误面，而不是死循环重建复杂树。
- 示例：固定“安全首页 + 错误码 + 重试”。

```c
void show_fallback_ui(const char *err)
{
    lv_obj_t *scr = lv_obj_create(NULL);
    lv_obj_t *lbl = lv_label_create(scr);
    lv_label_set_text_fmt(lbl, "UI异常恢复：%s", err);
    lv_obj_center(lbl);
    lv_scr_load(scr);
}
```

7. 输入/显示可观测性埋点
- 原则：打点记录“事件到绘制”的延迟分解，支持回放现场问题。
- 示例：轻量环形日志。

```c
typedef struct {
    uint32_t t_event;
    uint32_t t_flush_ready;
    uint16_t dirty_area;
} ui_trace_t;

void trace_flush(lv_area_t *area)
{
    g_trace[idx].t_event = lv_tick_get();
    g_trace[idx].dirty_area = (area->x2 - area->x1 + 1) * (area->y2 - area->y1 + 1);
}
```

8. 灾备级内存回收策略（自动降载）
- 原则：异常检测时触发“自动降载流程”，停止预加载与重建，保留核心功能。
- 示例：碎片/占用超阈值即触发回收。

```c
if (lv_mem_monitor.used_pct > 90 || lv_mem_monitor.frag_pct > 45) {
    stop_resource_prefetch();
    drop_non_critical_screens();
    disable_shadow_blur_animation();
    switch_to_low_memory_profile();
}
```

9. 任务优先级隔离与背压
- 原则：显示、IO、网络、传感、日志分开 task，且每层都有背压队列长度，避免上游压垮 UI。
- 示例：队列满时丢弃低优先级事件。

```c
if (uxQueueSpacesAvailable(ui_evt_q) == 0) {
    vTaskDelay(pdMS_TO_TICKS(1));
    // 或直接丢弃可合并事件，避免阻塞 UI 线程
    return;
}
```

10. 硬件回归样本化（边界场景）
- 原则：CI 之外保留一组固定“边界样本”用于每次发布前验证：亮度、FPS、内存、温漂条件。
- 示例：脚本化场景矩阵。

```bash
python tools/run_lvgl_hardware_suite.py --cases battery_low,low_brightness,max_brightness,cold_start
```

### 15.1 进阶生产策略检查清单
- [ ] 是否完成多 DPI/分辨率自适配，跨屏幕无明显重排抖动？
- [ ] 主题是否集中配置，热更新是否不引发大面积重建？
- [ ] 帧预算是否有效，是否有“超预算降级”路径？
- [ ] 按键/列表是否有防重入与幂等保护？
- [ ] 资源冷热路径是否按优先级加载，冷路径失败是否能回退？
- [ ] 是否有统一 fallback UI 与错误码提示？
- [ ] 能否导出事件-刷新延迟和脏区大小用于现场定位？
- [ ] 自动降载是否已定义阈值和恢复策略？
- [ ] 任务优先级和背压策略是否验证过优先级反转风险？
- [ ] 边界场景样本是否固定并持续运行？

### 16. 硬性代码修改规范（先分析，确认，诊断后再改）

在本项目中，对 LVGL/显示链路的代码改动必须遵守以下硬性流程。禁止“看到问题先改”，必须先做现象确认和根因闭环。

#### 16.1 先确认现象（Before Change）

1. 用户必须先确认的现象（最低要求）：
   - 设备/版本：`MCU / LCD 驱动芯片 / 显示接口`
   - 重现步骤：`3~5 步的可复现操作`（按钮、菜单路径、动画场景）
   - 触发频率：`必现 / 高频 / 偶现（给比例）`
   - 观察窗口：`开始→卡顿/黑屏→恢复所需时间`
   - 伴随信号：`日志、UART 输出、看门狗、温度/电量/亮度状态`

2. 采集证据后再谈优化点：
   - 现象截图/日志时间窗（含前后 5 秒）
   - 涉及屏幕参数（分辨率、刷新频率、缓冲区策略）
   - 相关计时值（`lv_timer_handler` 耗时、flush 间隔、队列深度、内存占用）

#### 16.2 根因分析规则（不可省）

- 先写“问题假设”再写“验证计划”，每个假设必须有可验证输入输出。
- 默认策略：
  1) 数据链路问题？
  2) 渲染调度问题？
  3) 帧缓冲与 DMA 问题？
  4) 主题/样式重算引发的重绘风暴？
  5) 资源/内存阈值触发？
- 如果任一假设证据不足，不得执行“策略性重构”；只做最小实验或加日志确认。

#### 16.3 未能立即定位原因时的日志要求（必须加）

先加最小诊断日志，不允许直接改逻辑。

```c
/* 示例：最小可观测埋点 */
typedef struct {
    uint32_t t_ms;
    uint32_t tick_now;
    uint32_t render_ms;
    uint32_t flush_ms;
    uint16_t dirty_px;
    int32_t  mem_used_kb;
    int32_t  mem_frag_pct;
} ui_diag_snapshot_t;

static void ui_diag_dump(const char *tag, const ui_diag_snapshot_t *s)
{
    // 用你项目统一日志接口替换下面输出
    LOG_I("UI-DIAG tag=%s t=%u tick=%u render=%u flush=%u dirty=%u mem=%uKB frag=%d%%",
          tag, s->t_ms, s->tick_now, s->render_ms, s->flush_ms,
          s->dirty_px, s->mem_used_kb, s->mem_frag_pct);
}

void ui_diag_before_commit(const char *tag)
{
    ui_diag_snapshot_t s = {0};
    s.t_ms = lv_tick_get();
    s.tick_now = xTaskGetTickCount();
    s.render_ms = lv_timer_get_next_timeout_ms();
    s.dirty_px = get_last_dirty_area_px();
    s.mem_used_kb = lv_mem_monitor.used_size / 1024;
    s.mem_frag_pct = lv_mem_test_fragmentation();
    ui_diag_dump(tag, &s);
}
```

#### 16.4 允许修改的边界（硬性约束）

- 允许做最小改动：`仅一个可复现问题点 + 可回滚代码块`。
- 禁止大规模结构性重构（除非经过 2 次失败改动后确认新瓶颈）。
- 每次改动必须有“回退路径”（feature flag、兼容分支、环境变量）。

#### 16.5 修改完成后的验收（Must Pass）

- 再现现象在本地和目标设备上各跑 1 次。
- 改前/改后对比：卡顿点、黑屏点、`lv_timer_handler` 长尾、flush 失败次数、内存碎片。
- 记录“是否支持回退”，并确认回退后现象消失。

#### 16.6 进阶生产策略检查清单（Hard Rule）
- [ ] 是否先拿到用户确认的现象与复现步骤？
- [ ] 是否形成了至少 2 个以上根因假设并逐一验证？
- [ ] 是否有完整时间线日志（触发前后至少 5 秒）？
- [ ] 无法立即判断原因时是否已先加最小日志再改？
- [ ] 修改是否限定单点（最小闭区间）且可回退？
- [ ] 改前改后对比是否量化（FPS/耗时/内存/flush）？
- [ ] 是否有“异常回退”策略（fallback）并已通过验证？

### 17. 运维友好优化（生产落地）

上一节是策略，这一节给“可直接上线”的实操清单。按序执行，优先从 1~3 开始。

1. 先做一次“无改动基线采样”
- 目的：给后续优化提供可比照基线。
- 示例：保留 10 秒性能日志快照。

```bash
python tools/lvgl_perf_probe.py --duration 10 --out out/baseline.json
```

2. 建立显示刷新“超时看门狗”
- 目的：避免 flush 卡死导致整屏静止。
- 示例：超过阈值触发自恢复。

```c
static uint32_t last_flush_ms = 0;
void on_flush_ready(lv_disp_t *disp)
{
    last_flush_ms = lv_tick_get();
    lv_disp_flush_ready(disp->driver->draw_buf);
}

void ui_watchdog_task(void *arg)
{
    while (1) {
        if ((lv_tick_get() - last_flush_ms) > 500) {
            LOG_E("UI watchdog timeout, trigger recovery");
            lv_timer_handler();  // 保留，便于日志记录点位
            request_ui_recover();
        }
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}
```

3. 统一“首屏最小化”渲染路径
- 目的：减少首屏首帧耗时和一次性对象风暴。
- 示例：先加载壳，再按需加载副组件。

```c
void build_home_root(lv_obj_t *parent)
{
    create_skeleton_layout(parent);     // 只保留基础骨架
    create_status_chip(parent);         // 常显核心区
    defer_expand_panels(parent);       // 非关键区域延后绘制
}
```

4. 防重复点击/重复跳页（幂等）
- 目的：防止一次点击触发双建树。

```c
static bool g_nav_lock;
static lv_obj_t *g_nav_current;

void safe_navigate(lv_obj_t *target)
{
    if (g_nav_lock || target == g_nav_current) return;
    g_nav_lock = true;
    lv_async_call(nav_request_cb, target);
}

void nav_request_cb(void *obj)
{
    g_nav_current = obj;
    lv_obj_del_async(lv_scr_act());
    lv_obj_t *scr = lv_obj_create(NULL);
    // load target page
    g_nav_lock = false;
}
```

5. 限速 `lv_obj_invalidate` 与 UI 状态抖动
- 目的：避免单帧重复触发导致 CPU 打满。

```c
static uint32_t last_ui_update_ms;
void invalidate_if_needed(lv_obj_t *obj)
{
    uint32_t now = lv_tick_get();
    if (now - last_ui_update_ms < 16) return;   // 60FPS 上限
    last_ui_update_ms = now;
    lv_obj_invalidate(obj);
}
```

6. 批量设置样式（减少内存与重排）
- 目的：用 style 对象替代每控件单独 set_style。

```c
LV_FONT_DECLARE(font16);
static lv_style_t style_title;

void init_title_style(void)
{
    lv_style_init(&style_title);
    lv_style_set_text_font(&style_title, &font16);
    lv_style_set_text_color(&style_title, lv_color_white());
    lv_style_set_pad_all(&style_title, 8);
}

void make_title(lv_obj_t *parent, const char *txt)
{
    lv_obj_t *l = lv_label_create(parent);
    lv_obj_add_style(l, &style_title, LV_PART_MAIN);
    lv_label_set_text(l, txt);
}
```

7. 避免在 `LV_EVENT_ALL` 用大回调
- 目的：复杂事件回调会放大每次输入开销。
- 建议：拆分事件类型，只监听 `LV_EVENT_CLICKED` / `LV_EVENT_VALUE_CHANGED`。

```c
static void on_btn(lv_event_t *e)
{
    if (lv_event_get_code(e) != LV_EVENT_CLICKED) return;
    // 只处理点击路径
}
```

8. 控制图层叠加复杂度
- 目的：透明 + 阴影 + 模糊叠加对性能冲击高。

```c
// 关键路径禁用复杂特性
lv_obj_set_style_bg_opa(obj, LV_OPA_COVER, 0);
lv_obj_set_style_shadow_width(obj, 0, 0);
lv_obj_set_style_blend_mode(obj, LV_BLEND_MODE_NORMAL, 0);
```

9. 一刀切的 fallback 模式
- 目的：异常时快速切回低负载显示，不让系统持续抖动。

```c
if (lv_mem_test_fragmentation() > 55 || fault_counter > 3) {
    set_low_framerate_mode(true);
    set_full_refresh_mode(false);
    disable_complex_animations(true);
}
```

2. 变更优先级与风险评分
- 策略：把落地顺序从“有经验猜测”改为“收益-成本-风险”量化。

3. 基线指标契约（Perf Contract）
- 策略：新增“改前/改后”唯一可比对指标，避免主观优化。
- 实施步骤：

  - 在章节末尾给出对比表模板。
- 由目标工程的设备或模拟器基准测试连续采集 30 秒指标，并产出 `baseline.json` 与 `after.json`；静态检查不能生成或替代这些数据。

4. 日志最小字段标准化（可观测性）
- 策略：统一日志字段，支持跨页面回溯。
- 实施步骤：
  - 使用统一字段名：`ts_ms/tick/reason/render_ms/flush_ms/dirty_px/mem_used_kb/frag_pct/page_id`。
  - 补齐 16.3 示例中的字段采集函数。
```c
LOG_I("UI-DIAG ts=%u tick=%u reason=%s render=%u flush=%u dirty=%u mem=%u frag=%d page=%u",

```
- 验收点：日志聚合能按字段筛选“场景+时间窗+页面”进行查询。

5. 故障注入与回归演练
- 策略：把常见异常变成固定演练脚本，避免修复只看“正常路径”。
- 实施步骤：
  - 新增演练项：flush 超时、输入风暴、低内存、重复导航、亮度突变。
  - 每项至少定义 `触发方式/预期现象/恢复期望`。
使用目标工程已有的日志、模拟器或硬件测试入口复现 flush 卡住、输入风暴、低内存、重复导航和亮度变化。
- 验收点：演练触发时能稳定复现异常并触发 fallback 或恢复策略。

6. 回退开关与降载一键化
- 策略：任何性能策略必须具备可立即禁用路径。
- 实施步骤：
  - 在配置层新增 runtime flag：`LVGL_OPT_LITE_MODE/LVGL_OPT_SAFE_MODE`。

7. RTOS 任务与队列预算
- 策略：为 UI 关键链路设置可执行的任务/队列约束，降低优先级反转。

- 验收点：首次可交互时延下降，且首页无空白闪烁。

### 19. 缺口补齐清单 V3（12项）

> 执行顺序按“低成本、高风险先收敛”排列。先补可观测、影响分析、故障注入，再推进版本化、快照、CI 分层等较重治理。

1. 错误可观测性标准化
- 策略：所有 checker、gate、脚本失败输出必须可机器读、可人工定位，禁止静默失败和模糊 `return`。
- 实施步骤：
  - 定义统一错误码：`E-SCHEMA-*`、`E-IO-*`、`E-THRESHOLD-*`、`E-CONFIG-*`。
  - 统一失败模板：`code/scope/file/expected/actual/action`。
  - 在脚本主入口集中捕获异常，并保留关键上下文摘要。
- 验收点：同一类故障重复运行 3 次，日志结构一致；CI 失败信息能直接定位到脚本、字段和建议动作。

2. 变更影响分析规范化
- 策略：规则、阈值、allowlist、平台映射变更前必须先说明影响面，避免“修一处破一片”。
- 实施步骤：
  - 每次变更记录：目标、影响平台、受影响 workflow、受影响 fixture、预期行为差异。
  - 对高风险变更保留 before/after 输入输出证据。
  - 明确是否影响 strict/warning、quality gate、route conflict、allowlist 行为。
- 验收点：缺少影响说明的变更不得进入 release gate；至少一条最小复现能证明行为差异。

3. 故障注入与模糊测试
- 策略：把字段缺失、类型错乱、空输出、权限失败等边界转成固定 fixture。
- 实施步骤：
  - 增加 `empty_output`、`missing_field`、`wrong_type`、`write_denied` 四类样例。
  - 高风险脚本各接入一个独立故障注入用例。
  - 对每个故障定义固定 exit code、错误码和关键错误文本。
- 验收点：触发故障时命令返回非 0，输出包含错误码和修复动作；正常 fixture 不受影响。

4. 规则元数据版本化
- 策略：policy、allowlist、schema、脚本 I/O 合同必须携带版本，避免隐式兼容造成误判。
- 实施步骤：
  - 配置文件增加 `version`、`schema_version`、`generated_by` 或 `owner` 字段。
  - 解析前校验最低兼容版本，不认识的主版本直接 fail。
  - 版本不兼容时输出迁移提示，而不是继续按旧逻辑解析。
- 验收点：旧字段缺失时给兼容错误；schema 主版本不匹配时阻断并说明期望版本。

5. 回归快照与工件标准化
- 策略：关键 gate 输出固定为稳定 JSON，减少“输出看似通过但语义漂移”。
- 实施步骤：
  - 统一字段顺序、计数格式、布尔字段和空数组输出。
  - 为关键 pass/fail 场景保存 snapshot。
  - 工件写入失败时 stdout 保留完整 JSON，进程返回非 0。
- 验收点：snapshot diff 能显示语义变化；无写权限目录下失败清晰且不吞 JSON。

6. 配置变更治理
- 策略：allowlist 和 policy 不能成为长期放宽质量的隐性入口。
- 实施步骤：
  - quick gate 启动时打印配置 md5、mtime、路径。
  - 增加空变更检查：只有格式变化、空白变化不得作为有效治理变更。
  - 变更说明必须包含新增/移除原因、过期条件和负责人。
- 验收点：空 diff 被拒绝；配置 hash 可在 CI 日志中追踪；allowlist 条目有明确退出条件。

7. 测试金字塔扩展
- 策略：用少量分层测试覆盖最大回归面，而不是只依赖端到端脚本。
- 实施步骤：
  - 单元层覆盖 parser、schema validator、阈值计算。
  - 集成层覆盖脚本入口、参数模式、artifact 写入。
  - 端到端层覆盖 CI 常用 fixture 套件。
- 验收点：修改 checker 时至少命中单元和集成测试；修改 gate 策略时必须命中端到端样例。

8. 文档与实现绑定
- 策略：文档条目必须能被执行和验证，禁止停留在泛化建议。
- 实施步骤：
  - 每条策略固定包含：触发条件、处理动作、边界/禁忌、回滚路径。
  - 为关键策略绑定实际命令、脚本或 fixture。
  - 增加常见误判案例和排除路径。
- 验收点：reviewer 能按文档独立复现一次问题并完成验证；无脚本入口的条目必须说明人工验收方式。

9. 平台能力映射一致性
- 策略：ESP32、STM32、JL、BK 是平台轴；FreeRTOS、Zephyr 是 RTOS 轴，两类能力不得混层。
- 实施步骤：
  - 平台能力统一描述显示、DMA、cache、NVS/flash、GPIO、时钟、电源、外设限制。
  - RTOS 能力统一描述 task、thread、queue、timer、ISR、watchdog、Kconfig/devicetree。
  - 新增平台或 RTOS 时先补 schema，再补映射，再接入 router。
- 验收点：任一平台或 RTOS 缺关键字段时校验失败；router 输出不会把 Zephyr 当平台或把 ESP32 当 RTOS。

10. 安全与权限边界加固
- 策略：工具输入路径、输出目录、临时文件和命令调用必须有最小权限边界。
- 实施步骤：
  - 对输入路径做基目录约束，拒绝路径穿越。
  - 写文件前检查目录存在、可写、非系统敏感目录。
  - 避免字符串拼接 shell 命令，优先使用结构化参数。
- 验收点：非法路径、无写权限、系统目录写入尝试均返回明确错误；不会产生 silent pass。

11. CI 质量门禁粒度化
- 策略：PR、main、release 使用不同严格度，避免本地开发被过度阻断，同时保证主干质量。
- 实施步骤：
  - 定义 `local/warn`、`pr/strict`、`main/strict`、`release/blocking` 四档行为。
  - 增加环境变量默认策略：CI 环境默认 strict，本地默认 warning。
  - gate 输出中明确当前模式、阈值、实际计数和失败原因。
- 验收点：同一输入在 warning 与 strict 下行为差异可预测；CI 日志能看到最终生效模式。

12. 交付闭环任务化
- 策略：每次审查输出必须落成下一步可执行任务，而不是只给建议。
- 实施步骤：
  - 每项输出固定包含：`Next Action`、`Owner`、`Priority`、`Expected Metric`。
  - 高风险项必须带验证命令或 fixture 名。
  - 每次版本收口更新一次状态：open/doing/done/deferred。
- 验收点：审查结论能直接转为任务清单；2-3 周内能按指标关闭或明确延期原因。
